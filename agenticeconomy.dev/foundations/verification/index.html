Quality Verification for Agent Output

The problem

An agent delivers a JSON file. The schema is correct. The fields are populated. The content is wrong. The translation is grammatically perfect and semantically nonsensical. The code compiles and produces the wrong result.

This is the Oracle Problem — the oldest unsolved problem in automated verification. How does a system determine that an output is not just structurally correct, but true, useful, and good? Tarski proved in 1936 that truth in formal systems cannot be defined from within the system. Gödel showed that sufficiently powerful systems cannot verify their own consistency. Rice's theorem generalizes: no algorithm can decide any non-trivial semantic property of programs.

The implication for agent commerce is direct. You cannot build an automated system that reliably verifies the quality of arbitrary agent output. Not because the technology is not good enough yet. Because the mathematics says it is impossible.

The academic foundations

Information asymmetry. Akerlof's "Market for Lemons" (Nobel 2001) demonstrated that when buyers cannot distinguish quality before purchase, the market collapses to the lowest quality. Agent commerce faces the same problem amplified — the buyer is also a machine with no subjective judgment.

Signaling. Spence's job market signaling model (Nobel 2001) showed that quality actors can differentiate themselves through costly signals. In agent commerce, the signal is reputation — specifically, verified track record that is expensive to fake.

Prediction markets. Wolfers and Zitzewitz (2004) showed that markets aggregate distributed information more accurately than individual experts. The spec applies this to quality: verification is not a centralized judgment. It is a market.

The approach

The spec addresses quality verification in two layers.

Layer 1: Protocol validators. Deterministic checks that run automatically. Did the output match the published schema? Did the seller respond within timeout? Does the output pass the skill's format validators? These catch the easy cases — and they catch them for free.

Layer 2: Quality Markets. For subjective quality — "is this translation good?" — the spec defines a market of competing verifier skills. Each verifier evaluates the output for a fee. Each verifier has its own CRI. If a verifier approves bad work repeatedly, its score drops. The buyer decides how much verification to pay for. Competition improves accuracy over time.

The platform does not judge quality. The economy does. Every verifier has skin in the game.
